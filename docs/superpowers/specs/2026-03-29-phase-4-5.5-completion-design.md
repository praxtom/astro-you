## 2. Panchang into Proactive Triggers + Personalized Evening Reflection

**What:** Wire `usePanchang` data into `useProactiveTriggers` for Rahu Kaal alerts and transit-aware evening reflections.

**Files to modify:**
- `src/hooks/useProactiveTriggers.ts` — add 2 new trigger types

**New Trigger A — Rahu Kaal Warning:**
- **Window:** Check every 5 min (existing poll interval)
- **Logic:** Parse `panchang.rahu_kaal` string (e.g., "10:30 AM - 12:00 PM"), check if current time falls within +-15 min of start
- **TriggerKey:** `rahu_kaal_{today}`
- **Message:** "Rahu Kaal begins soon. Avoid starting new ventures for the next 90 minutes. Use this time for reflection."
- **Toast type:** Warning (amber)

**New Trigger B — Personalized Evening Reflection (replaces generic gratitude):**
- **Window:** 9 PM - 11 PM (same as existing Evening Gratitude)
- **Logic:** Replace the static "Before you rest, what are you grateful for today?" with a transit-aware message
- **Data needed:** `panchang.tithi` and today's dominant transit from existing transit context
- **Message template:** "Today's {tithi} energy asked you to {tithi_guidance}. How did you navigate that?"
- **Tithi guidance mapping:** Simple lookup — Pratipada → "begin fresh", Purnima → "reflect on fullness", Amavasya → "turn inward", etc.
- **Fallback:** If panchang data unavailable, use original generic message

**Panchang data source:** The existing `usePanchang` hook fetches from `/api/kundali?chartType=PANCHANG`. Since `useProactiveTriggers` runs in Dashboard context where panchang is already fetched, we pass panchang data as a parameter rather than re-fetching.

**Hook signature change:**
```typescript
export function useProactiveTriggers(panchangData?: PanchangData): void
```

## 3. Shared Firestore Cache (Panchang)

**What:** Cache daily Panchang in a shared Firestore collection so multiple users don't each call the API.

**Files to modify:**
- `src/hooks/usePanchang.ts` — check cache first, write-through on miss
- `firestore.rules` — add shared collection rules
- `netlify/functions/kundali.ts` — write to shared collection on PANCHANG requests (server-side)

**Firestore structure:**
```
/shared/panchang/{YYYY-MM-DD}
  ├── tithi: string
  ├── nakshatra: string
  ├── yoga: string
  ├── karana: string
  ├── rahu_kaal: string
  ├── city: string
  ├── createdAt: Timestamp
  └── rawData: object (full API response)
```

**Cache strategy:**
1. `usePanchang` hook → check `/shared/panchang/{today}` in Firestore
2. If exists and `city` matches → return cached data (no API call)
3. If miss → call `/api/kundali?chartType=PANCHANG` as before
4. Server-side: `kundali.ts` writes result to `/shared/panchang/{today}` after successful API call
5. Location-specific: Cache key includes city for accuracy (Rahu Kaal varies by location)

**Updated Firestore structure (location-aware):**
```
/shared/panchang/{YYYY-MM-DD}_{city_slug}
```

**Security rules:**
```
match /shared/{document=**} {
  allow read: if request.auth != null;
  allow write: if false; // Server-side only via Admin SDK
}
```

**Server-side write:** The `kundali.ts` Netlify function uses Firebase Admin SDK (already available) to write to the shared collection after fetching from the astrology API.

## 6. Daily Synastry Alerts (Compassion Nudges)

**What:** Trigger empathetic relationship alerts based on comparing user's transits with partner's natal chart.

**Files to modify:**
- `src/hooks/useProactiveTriggers.ts` — add synastry alert trigger

**Trigger logic:**
- **Window:** 2 PM - 5 PM (once per day)
- **Condition:** `keyRelationships` has at least one member with birth data (`birthData` present)
- **TriggerKey:** `synastry_alert_{today}`
- **Action:** Call `/api/proactive-nudge` with `triggerType: 'synastry_alert'` and include the first partner's birth data + user's birth data
- **Backend:** `generateProactiveNudge()` in `gemini.ts` already supports custom trigger types. It receives both charts and generates an empathetic nudge.

**Nudge prompt guidance (added to `generateProactiveNudge`):**
```
For synastry_alert: Compare the user's current transits with their partner's natal chart.
Focus on emotional friction points (Moon aspects), communication blocks (Mercury),
and passion dynamics (Mars/Venus). Tone: compassionate, not alarming.
Example: "Your partner may feel more sensitive today. A small gesture of kindness goes far."
Never use fear language. Frame as opportunity for deeper connection.
```

**Relationship selection:** Use the first relationship with `relation === 'partner'`, or if none, the first relationship with birth data available.

## 9. Shareable Chart Images

**What:** Share modal with download + social sharing using existing `captureChartAsPng()`.

**Files to create:**
- `src/components/ChartShareModal.tsx` — modal with share options

**Files to modify:**
- `src/pages/Synthesis.tsx` — add share button next to chart download button
- `src/pages/Dashboard.tsx` — add share button on Kundali card
- `src/pages/Compatibility.tsx` — add share button on synastry results

**ChartShareModal props:**
```typescript
interface ChartShareModalProps {
  isOpen: boolean;
  onClose: () => void;
  elementId: string;        // DOM element to capture
  title: string;            // e.g., "My Birth Chart"
  description?: string;     // e.g., "Sun in Aries, Moon in Cancer"
}
```

**Share flow:**
1. User clicks share button → modal opens
2. Chart is captured via `captureChartAsPng(elementId)` → preview shown in modal
3. Options:
   - **Download** — triggers `downloadChart()` (existing function)
   - **Copy Link** — uploads to Firebase Storage via `uploadChartToStorage()`, copies public URL
   - **WhatsApp** — `https://wa.me/?text={encodedText}%20{publicUrl}`
   - **Twitter/X** — `https://twitter.com/intent/tweet?text={encodedText}&url={publicUrl}`
   - **Web Share API** — if `navigator.share` available (mobile browsers), use native share sheet

**Branding:** The captured image should include a small "AstroYou" watermark in the bottom-right corner. Add this as an overlay div that's only visible during capture (hidden in normal UI).

**Public URL generation:** Uses existing `uploadChartToStorage()` which returns a Firebase Storage download URL. These URLs are already publicly accessible via storage rules.
